# Конфигурация

Все настройки `telemt-shaper` хранятся в `/opt/telemt-shaper/config.py`. Любой параметр опционален: что не указано — берётся из дефолтов скрипта.

После изменений:

```bash
sudo systemctl restart telemt-shaper
```

Для проверки, что конфиг читается правильно, посмотри лог при запуске — там видно итоговые значения.

## Параметры

### `IFACE`

```python
IFACE = "ens18"
```

Сетевой интерфейс, на котором работает шейп. Должен быть тот же, через который ходит трафик MTProto-прокси (обычно интерфейс с дефолтным маршрутом).

Посмотреть список:
```bash
ip -br link
```

Дефолт: `"eth0"` (но установщик подставляет реальный при первой установке).

---

### `SHAPE_LEVELS`

```python
SHAPE_LEVELS = [
    {"threshold_mbps": 60, "limit_mbps": 40, "exceed_ticks": 2},  # L0
    {"threshold_mbps": 36, "limit_mbps": 24, "exceed_ticks": 4},  # L1
    {"threshold_mbps": 20, "limit_mbps": 8,  "exceed_ticks": 4},  # L2
]
```

Уровни шейпа. Применяются по порядку:
- **L0** — первый шейп (для нешейпленного IP, превысившего порог)
- **L1** — апгрейд с L0, если клиент продолжает упираться
- **L2** — апгрейд с L1
- ...можно добавлять сколько угодно уровней

Поля:

| Поле | Что значит |
|------|------------|
| `threshold_mbps` | Порог: превышение в течение `exceed_ticks` тиков → переход на этот уровень |
| `limit_mbps` | Лимит, который применится при попадании на этот уровень |
| `exceed_ticks` | Сколько тиков подряд (по `CHECK_INTERVAL` секунд) держать выше threshold |

**Правило хорошего конфига:** `threshold_mbps` уровня N+1 должен быть **ниже 90%** от `limit_mbps` уровня N. Иначе клиент физически не сможет достичь нового порога — он уже ограничен предыдущим уровнем, а реальная скорость на TCP payload получается ~90-95% от cake bandwidth.

При старте скрипт сам валидирует это правило и пишет предупреждения:

```
WARNING: L1.threshold (40) >= L0.limit (40) — апгрейд на L1 НИКОГДА не сработает.
         Рекомендуется L1.threshold <= 36
```

#### Один уровень — тоже валидно

Если хочется простой шейп без апгрейдов — оставь один элемент:

```python
SHAPE_LEVELS = [
    {"threshold_mbps": 50, "limit_mbps": 10, "exceed_ticks": 3},
]
```

Поведение: превысил 50 Мбит/с три раза подряд → урезали до 10 Мбит/с. Никаких апгрейдов, только снятие после cooldown.

#### Сколько тиков выбирать

`CHECK_INTERVAL × exceed_ticks` = время реакции в секундах.

- `exceed_ticks=2` (10 секунд) — реактивно, ловит даже короткие злоупотребления, но больше шансов на ложные срабатывания
- `exceed_ticks=4` (20 секунд) — сбалансированно
- `exceed_ticks=6` (30 секунд) — мягко, реагирует только на устойчивую нагрузку

---

### `CALM_RATIO`

```python
CALM_RATIO = 0.5
```

Доля от текущего лимита, ниже которой клиент считается «успокоившимся».

- `0.5` (дефолт): на L0 (limit 40) успокоился если упал ниже 20 Мбит/с
- `0.3`: более строго — успокоился только если ниже 30% (12 Мбит/с на L0)
- `0.8`: либерально — успокоился даже при 32 Мбит/с

Чем выше значение, тем быстрее снимается шейп. Чем ниже — тем дольше клиент должен «молчать», чтобы получить разморозку.

---

### `COOLDOWN_SECS`

```python
COOLDOWN_SECS = 120
```

Сколько секунд держать «успокоился» подряд, чтобы снять шейп. Если за это время скорость поднимется выше calm-порога — отсчёт обнулится.

- `60` — быстро отпускаем (1 минута)
- `120` (дефолт) — стандартно (2 минуты)
- `300` — строго (5 минут)

---

### `CHECK_INTERVAL`

```python
CHECK_INTERVAL = 5
```

Интервал между проверками в секундах.

- `5` (дефолт) — стандарт. На прокси с 1000 активных сокетов даёт ~0.05s CPU на тик.
- `3` — реактивнее, но больше нагрузка на ss и больше шанс «дёргания» при коротких всплесках
- `10` — спокойнее, экономнее, но дольше реакция

Не рекомендую ставить меньше 3 — `ss` начнёт заметно потреблять CPU.

---

### `UPGRADE_HEADROOM`

```python
UPGRADE_HEADROOM = 0.9
```

Используется только для валидации конфига при старте. Если порог следующего уровня выше `UPGRADE_HEADROOM × лимит предыдущего` — пишется warning.

Менять обычно не нужно. Если ты уверен в значениях и хочешь убрать шум — поставь `0.99` или `1.0`.

---

### `SKIP_NETWORKS_RAW`

```python
SKIP_NETWORKS_RAW = [
    "91.108.4.0/22",  # Telegram
    "10.0.0.0/8",     # private
    # ... ещё много
    "203.0.113.0/24", # своя добавка
]
```

CIDR-список сетей, которые шейпер **не трогает**. По умолчанию включены:
- Все официальные диапазоны Telegram (с [core.telegram.org](https://core.telegram.org/resources/cidr.txt))
- RFC1918 (приватные сети)
- 127.0.0.0/8, 169.254.0.0/16

Если хочешь добавить свои сети (например, IP админов или мониторинг) — переопредели весь список в `config.py`. Полный дефолтный список можно скопировать из `telemt-shaper.py` (переменная в `_DEFAULTS`).

---

### Пути к файлам

```python
LOG_FILE = "/var/log/telemt-shaper.log"
PID_FILE = "/var/run/telemt-shaper.pid"
LOG_MAX_BYTES = 10 * 1024 * 1024
LOG_BACKUP_COUNT = 3
```

Обычно менять не нужно. `LOG_BACKUP_COUNT` — сколько ротированных файлов хранить (`telemt-shaper.log.1`, `.log.2`, и т.д.).

---

### `STATE_TTL_SECS`

```python
STATE_TTL_SECS = 600
```

TTL для записей в словарях `ip_last_seen` и `exceed_count`. IP, не появлявшиеся в `ss` в течение этого времени, удаляются из памяти. Шейпленные IP под TTL не подпадают — их состояние всегда живое.

Менять не нужно, разве что хочется тоньше отслеживать «возвращающихся» клиентов.

---

### `FILTER_PRIO`

```python
FILTER_PRIO = 10
```

Приоритет tc-фильтров. Меняй только если у тебя на интерфейсе уже есть другие фильтры с тем же prio и они конфликтуют. По умолчанию — 10.

## Примеры конфигов под разные сценарии

### Мягкий — для домашнего сервера на 1-2 человек

Шейпит только тех, кто реально гонит много трафика, и довольно либерально.

```python
SHAPE_LEVELS = [
    {"threshold_mbps": 100, "limit_mbps": 60, "exceed_ticks": 4},
    {"threshold_mbps": 50,  "limit_mbps": 25, "exceed_ticks": 6},
]
COOLDOWN_SECS = 60
```

### Стандартный — дефолт

Из коробки, проверен в реальной эксплуатации.

```python
SHAPE_LEVELS = [
    {"threshold_mbps": 60, "limit_mbps": 40, "exceed_ticks": 2},
    {"threshold_mbps": 36, "limit_mbps": 24, "exceed_ticks": 4},
    {"threshold_mbps": 20, "limit_mbps": 8,  "exceed_ticks": 4},
]
```

### Жёсткий — для перегруженного публичного прокси

Быстро ловит и сильно режет.

```python
SHAPE_LEVELS = [
    {"threshold_mbps": 30, "limit_mbps": 15, "exceed_ticks": 2},
    {"threshold_mbps": 12, "limit_mbps": 5,  "exceed_ticks": 3},
    {"threshold_mbps": 4,  "limit_mbps": 1,  "exceed_ticks": 4},
]
COOLDOWN_SECS = 300
CALM_RATIO = 0.3
```

### Только один уровень — простой шейп без апгрейдов

```python
SHAPE_LEVELS = [
    {"threshold_mbps": 50, "limit_mbps": 10, "exceed_ticks": 3},
]
```
